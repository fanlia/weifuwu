# Org — 商业价值分析

> 记录于 2025-07-17 的产品讨论。
> 核心问题：Org 帮谁解决什么实际问题？

---

## 一、Org 解决的三个断层

```
断层1：工具断层
  团队用 ChatGPT / Claude / Kimi / 豆包 ...
  → 知识散落在各个平台
  → 无法沉淀到团队内部

断层2：协作断层
  人和 AI 是"使用关系"，不是"协作关系"
  → AI 给个答案，人复制粘贴到群里
  → 没有上下文，没有工作流，没有审批

断层3：架构断层
  想自建 AI 平台 → 要搭用户系统、聊天系统、知识库、Agent
  → 每个 SaaS 团队都在重复造轮子
  → weifuwu 解决了"造轮子"的问题，但还缺一个"开箱即用的产品"
```

Org 要解决的问题：**不是 AI 能不能回答，而是 AI 怎么融入团队、怎么有上下文、怎么执行、怎么沉淀、怎么管理。** 这是企业用 AI 的"最后一公里"。

## 二、对四类用户的具体价值

### 🧑‍💼 企业老板 / CTO（决策者）

**他们的真实烦恼：**
- 公司买了 ChatGPT Team，但员工还是各用各的，知识散得到处都是
- 不知道花了多少钱、用在哪里、效果如何
- 员工把公司代码贴到公共 AI 平台，存在数据泄露风险
- 想推广 AI 但推不动，团队说"不知道能帮我做什么"

**Org 的解法：**

| 痛点 | 解法 |
|------|------|
| AI 支出黑洞 | 所有 AI 调用走 Org 的 Agent，统一计费、统一审计，一个 Tenant 一个 API Key |
| 知识沉淀不了 | 所有对话发生在 Department 里，AI 的回答自动沉淀到部门 |
| 数据安全 | Org 自部署，数据全在你的服务器上，不经过第三方 |
| 推广难 | Agent 是部门成员，@一下就能用，零学习成本 |

### 🧑‍💻 一线员工（实际使用者）

**他们的真实烦恼：**
- 每次都要：打开 ChatGPT → 写 prompt → 复制答案 → 粘贴到群里 → 同事问"你用的什么模型" → 再解释一遍
- 每次问 AI 都要重新描述背景，AI 记不住之前的事
- 不知道 AI 的信息来源，不敢直接用
- AI 给个方案，但没人审批，不敢执行

**Org 的解法：**

| 痛点 | 解法 |
|------|------|
| 来回切换 | 直接在部门聊天里 @AI 机器人，不离开聊天界面 |
| 上下文丢失 | AI Agent 能看到部门历史消息和知识库，不需要重复交代 |
| 答案不可信 | Agent 回答带知识库引用（RAG），点一下看到原始文档 |
| 不敢执行 | Human-in-the-loop：AI 说"要发邮件给客户，确认吗？"，你点同意才执行 |

### 🤖 AI Agent（数字同事）

**Agent 的"痛点"（拟人化）：**
- 问"我们公司最近的销售数据"，但不知道"我们"是谁
- 能推理但执行不了，只能说"建议你用这个 API"
- 每次都是独立对话，没有团队归属感

**Org 的解法：**

| 痛点 | 解法 |
|------|------|
| 没有上下文 | Agent 绑定到 Department，知道自己在哪个团队、能访问哪些知识库 |
| 没有工具 | Agent 有 tool calling，可以直接调公司内部 API（查 CRM、发邮件、建工单）|
| 没有身份 | Agent 是 Department 的固定成员，新人进部门就能看到历史 AI 对话 |

**核心观点：Agent 不是工具，是一个有身份、有权限、有工具的"数字同事"。**

### 🏢 SaaS 创业者（用 Org 做产品）

**他们的真实烦恼：**
- 想搭一个 AI SaaS 卖给企业客户，但用户系统、聊天、知识库、Agent 都要从头写
- 团队只有 3 个人，做不完

**Org 的解法：**

| 痛点 | 解法 |
|------|------|
| 重复造轮子 | weifuwu 的 `user()` 直接搞定用户系统 |
| Multi-tenant 很难 | Org 的 Tenant 层天然支持数据隔离、配置隔离 |
| 实时消息基建重 | weifuwu 的 `messager()` 封装了 WebSocket、离线消息、已读回执 |
| AI 集成门槛高 | weifuwu 的 `agent()` + `kb()` 一条龙搞定 LLM + RAG + tool calling |

**一句话：3 个人的团队也能交付企业级 AI 产品，因为基础设施已经搭好了。**

## 三、用户一天的工作流对比

| 时间段 | 不用 Org | 用 Org |
|--------|---------|--------|
| 09:00 | 打开 Slack / 飞书 看消息 | 打开 Org 看部门消息 |
| 09:30 | 查合同条款 → 打开知识库系统 → 搜索 → 找不到 → 问法务 | @合同助手（AI Agent）→ 自动检索知识库 → 带引用的答案 |
| 10:00 | 收到客户需求 → 打开 ChatGPT 写方案 → 复制到飞书文档 | @方案助手 → AI 流式输出方案 → 同事在线评论 |
| 14:00 | 想查上月销售数据 → 问销售主管 → 等待回复 | @数据助手 → AI 调 CRM API → 直接返回数据 |
| 16:00 | AI 给了个建议 → 不确定对不对 → 先放着 | AI 建议需要审批 → 主管点同意 → AI 自动执行 |

## 四、目标客户画像

### Tier 1 — 最匹配（PMF 最可能）

| 画像 | 痛点 | 为什么选 Org |
|------|------|-------------|
| **AI SaaS 创业公司**（5-20 人） | 想快速搭建 AI 协作产品卖给客户，不想从头写用户/消息/知识库 | Org 直接是 multi-tenant SaaS 架构，weifuwu 全套模块开箱即用 |
| **企业内部 AI 平台团队** | 各部门都在用不同的 AI 工具，数据孤岛，管理混乱 | Org 的 tenant → company → department 天然适合大型组织 |

### Tier 2 — 可扩展

| 画像 | 场景 |
|------|------|
| **咨询 / 律所 / 会计师事务所** | 知识密集型行业，每个客户是一个 tenant，每个项目是一个 department，AI Agent 辅助检索和写作 |
| **远程团队 / 数字游民公司** | 团队分布全球，AI Agent 作为 24 小时在线同事处理异步工作 |
| **教育机构** | 每个班级一个 department，AI 助教答疑，知识库放课件 |

## 五、商业模式

```
模式 A：卖产品（SaaS）
  - 客户在你的云上部署 Org
  - 按 tenant 数 / Agent 调用量 / 存储量收费
  - 类似 Slack + Notion AI 的定价

模式 B：卖源码（White-Label）
  - 客户拿到 Org 源码，自己部署和定制
  - 适合大企业、政府、金融机构（数据合规要求高）
  - 类似 Supabase 的 self-hosted 模式

模式 C：卖框架（间接）
  - Org 作为 weifuwu 的最佳实践
  - 客户看了 Org 说"我要定制" → 买 weifuwu 自己搭
  - 类似 Vercel 的 demo 项目引导客户用框架
```

## 六、竞争定位

| 对比对象 | 他们的做法 | Org 的优势 |
|---------|----------|-----------|
| **Slack + AI 插件** | 聊天工具上叠 AI，但知识库和 Agent 是外挂 | Org 原生内置 Agent + 知识库，不需要拼凑 |
| **Notion AI** | 文档里写 prompt，但缺乏实时协作和结构化组织 | Org 有 department 聊天 + Agent 实时交互 |
| **钉钉 / 飞书 AI 助手** | 封闭生态，不能自定义 Agent | Org 开源可控，Agent 可编程、可接入任意模型 |
| **Dify / Coze** | AI 应用搭建平台，但不是协作平台 | Org 是"协作平台 + AI 能力"，不是"AI 工具" |
| **LangChain 生态** | 开发者工具，不是产品 | Org 是开箱即用的产品，不需要写代码 |

**核心差异：** Org 是一个 **Enterprise AI Mesh**，不是又一个 AI 聊天工具。它把人、AI、知识、消息放在同一个组织架构下。

## 七、窗口期分析（为什么是现在？）

2024-2025 年是"企业 AI 协作"的窗口期：

- **DeepSeek 等国产模型成熟** — 推理成本断崖式下降，企业愿意用 AI 做更多事
- **RAG 技术成熟** — PGVector 级别的向量数据库已足够满足 80% 场景
- **企业从"尝鲜"转向"落地"** — 2023 年试用 ChatGPT，2024-2025 年要的是"部署到团队里"
- **开源自建需求爆发** — 数据安全、合规、成本控制，大企业不想用 SaaS 版 AI 工具

但市场上还没有一个 "开源的企业 AI 协作平台"。Dify 偏 AI 应用搭建，ChatGPT Team 是闭源 SaaS，Slack AI 是插件。**Org 的 niche 是"开源 + 可部署 + 企业组织架构 + AI 原生"。**

## 八、风险与挑战

| 风险 | 缓解策略 |
|------|---------|
| 和现有聊天工具竞争（Slack / 飞书） | 不竞争"通用聊天"，只做"AI 协作"垂直场景 |
| 开源项目冷启动难 | 先作为 weifuwu 的官方 demo 发布，借框架用户基础 |
| AI Agent 质量依赖模型能力 | 设计为 model-agnostic，用户自己选模型 |
| Multi-tenant 运维复杂度 | weifuwu 已经做了 postgres + redis + queue 的封装 |
